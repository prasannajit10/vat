OWASP Top 10 Reference Guide

A01

Broken Access Control

Critical

Access control enforces policy so users cannot act outside their intended permissions. Failures lead to unauthorized data disclosure, modification, or destruction. In 2025 this also absorbs SSRF attacks from the prior list.

Risk Factors

  • Bypassing access control checks by modifying URLs or API requests
  • Insecure Direct Object References (IDOR)
  • Missing access controls for POST, PUT, DELETE methods
  • Elevation of privilege â€” acting as admin when logged in as user
  • Server-Side Request Forgery (SSRF) â€” now merged into this category

Attack Examples

GET /api/users/1234/account → change to /api/users/5678/account

Attacker modifies the user ID parameter to access another user's account data. SSRF payloads (e.g., fetching internal IPs) are also covered here.

Remediation

  • Implement server-side access control; deny by default
  • Enforce record ownership — don't rely on user-supplied IDs
  • Validate and allowlist all user-supplied URLs (mitigate SSRF)
  • Log access control failures; alert admins on repeated failures
A02

Security Misconfiguration

Critical

Security misconfiguration is the most commonly seen issue. It occurs when security settings are not defined, implemented, or maintained properly — including missing hardening, open cloud storage, misconfigured HTTP headers, or verbose error messages.

Risk Factors

  • Unnecessary features enabled (ports, services, pages, privileges)
  • Default accounts and passwords unchanged
  • Overly informative error messages exposing stack traces
  • Missing security headers (CSP, X-Frame-Options, HSTS)
  • Cloud services with overly permissive access policies

Attack Examples

Server: Apache/2.4.41 (Ubuntu) | X-Powered-By: PHP/7.2.0 Error: PDOException in /var/www/app/db.php on line 42

Exposed server version info and stack traces allow targeted CVE exploitation and path mapping.

Remediation

  • Implement a repeatable hardening process across all environments
  • Remove unused features, components, documentation, and samples
  • Automate configuration verification with security benchmarks
  • Segment architecture and use proper permissions on cloud services
A03

Software Supply Chain Failures

Critical

A new category expanding on "Vulnerable Components." It covers the entire software ecosystem — dependencies, build systems, CI/CD pipelines, and distribution infrastructure. Attacks like dependency confusion and malicious packages belong here.

Risk Factors

  • Using components with known vulnerabilities (CVEs)
  • No Software Bill of Materials (SBOM) or dependency tracking
  • Compromised CI/CD pipelines allowing malicious code injection
  • Dependency confusion / typosquatting attacks on package registries
  • Lack of signature verification on packages and updates

Attack Examples

npm install @company/utils â† attacker publishes "company-utils" on public npm // Malicious post-install script exfiltrates .env, SSH keys, tokens

Dependency confusion attack: malicious public package shadows a private internal package, running arbitrary code at install time.

Remediation

  • Maintain a Software Bill of Materials (SBOM) and audit regularly
  • Pin dependency versions and verify checksums/signatures
  • Secure CI/CD pipelines with least-privilege and signed artifacts
  • Use private package registries and namespace scoping
A04

Cryptographic Failures

High

Failures related to cryptography which often lead to exposure of sensitive data. This includes use of weak algorithms, insufficient key management, or transmission of data in cleartext.

Risk Factors

  • Transmitting data in cleartext (HTTP, SMTP, FTP)
  • Using deprecated cryptographic algorithms (MD5, SHA1, DES)
  • Weak key generation or reuse of cryptographic keys
  • Missing TLS enforcement or improper certificate validation

Attack Examples

passwords stored as: MD5("password123") → 482c811da5d5b4bc...

Unsalted MD5 hashes are trivially cracked using rainbow tables, exposing all user passwords.

Remediation

  • Classify data and apply controls based on sensitivity
  • Use strong hashing with salt (bcrypt, scrypt, Argon2)
  • Enforce HTTPS everywhere with HSTS
  • Use authenticated encryption (AES-256-GCM)
A05

Injection

High

Injection flaws occur when untrusted data is sent to an interpreter as part of a command or query. Hostile data can trick the interpreter into executing unintended commands or accessing data without authorization.

Risk Factors

  • User-supplied data not validated, filtered, or sanitized
  • Dynamic queries or non-parameterized calls without escaping
  • Hostile data used in ORM search parameters
  • SQL, NoSQL, OS, LDAP injection vectors

Attack Examples

SELECT * FROM users WHERE name = '' OR '1'='1' --'

A classic SQL injection that bypasses authentication by making the WHERE clause always true.

Remediation

  • Use parameterized queries / prepared statements
  • Apply server-side input validation with allowlists
  • Escape special characters for the specific interpreter
  • Use LIMIT and other SQL controls to prevent mass disclosure
A06

Insecure Design

High

Insecure design represents weaknesses in the design and architecture of an application. It focuses on risks related to design flaws as opposed to implementation bugs. Even perfect implementation cannot fix an insecure design.

Risk Factors

  • Missing or ineffective threat modeling during design phase
  • No secure design patterns or reference architectures used
  • Business logic flaws allowing abuse scenarios
  • Missing rate limiting on critical business flows

Attack Examples

POST /checkout â€” no limit on quantity or price validation {"item":"premium_plan","price":0.01,"qty":999}

Business logic flaw: API trusts client-side price, allowing purchase of premium items for pennies.

Remediation

  • Establish a secure development lifecycle with threat modeling
  • Use secure design patterns and paved-road architectures
  • Implement plausibility checks and business-logic validation server-side
  • Write unit and integration tests for abuse/misuse stories
A07

Authentication Failures

Critical

Weaknesses in authentication and session management allow attackers to compromise passwords, keys, or session tokens, or exploit implementation flaws to assume other users' identities temporarily or permanently.

Risk Factors

  • Permits brute force or credential stuffing attacks
  • Allows default, weak, or well-known passwords
  • Missing or ineffective multi-factor authentication (MFA)
  • Session IDs exposed in URLs or not rotated after login

Attack Examples

POST /login â€” 10,000 attempts with rockyou.txt wordlist No lockout, no CAPTCHA, no rate limit â†’ password "qwerty123" found

Credential stuffing using leaked password databases against a login endpoint with no brute-force protection.

Remediation

  • Implement multi-factor authentication (TOTP, WebAuthn)
  • Enforce strong password policies; check against breached databases
  • Limit failed login attempts with progressive lockout
  • Use secure, server-side session management with random session IDs
A08

Software & Data Integrity Failures

High

Software and data integrity failures relate to code and infrastructure that does not protect against integrity violations â€” including using software from untrusted sources, insecure CI/CD pipelines, and unsafe deserialization.

Risk Factors

  • Application relies on plugins or libraries from untrusted CDNs without SRI
  • Insecure CI/CD pipeline allowing unauthorized code modifications
  • Auto-update functionality without integrity verification
  • Unsafe deserialization of untrusted data leading to RCE

Attack Examples

<script src="https://cdn.example.com/lib.js"> // No integrity attribute â€” compromised CDN serves malicious code

External script loaded without Subresource Integrity (SRI) â€” CDN compromise leads to supply chain attack on all users.

Remediation

  • Use Subresource Integrity (SRI) for all external resources
  • Verify digital signatures on software and updates
  • Implement code review and approval workflows in CI/CD
  • Avoid unsafe deserialization; use safe formats like JSON
A09

Security Logging & Alerting Failures

Medium

Renamed from "Logging & Monitoring Failures" to emphasize effective alerting. Without proper logging, alerting, and active response, breaches cannot be detected and attackers maintain persistence, extract data, or pivot to other systems.

Risk Factors

  • Auditable events not logged (logins, failed logins, transactions)
  • Logs stored locally without centralization or backup
  • No automated alerting thresholds or escalation processes
  • Penetration testing and scanner activity does not trigger alerts

Attack Examples

Attacker exfiltrates 3.2M records over 194 days â€” zero alerts triggered (Average breach dwell time â€” IBM Security 2024)

Without proper logging and alerting, attackers remain undetected for months, exfiltrating data at will.

Remediation

  • Log all authentication, access control, and input validation failures
  • Send logs to centralized SIEM with automated alerting rules
  • Establish effective real-time alerting for suspicious activities
  • Create and regularly test incident response and recovery plans
A10

Mishandling of Exceptional Conditions

Medium

A new category for 2025 highlighting the significance of secure error handling and system resilience. Applications that fail to properly handle unexpected inputs, resource exhaustion, or edge cases can crash, leak data, or become exploitable.

Risk Factors

  • Uncaught exceptions exposing stack traces, file paths, or DB queries
  • Missing error boundaries causing cascading application failures
  • Improper handling of null, empty, or oversized inputs
  • Resource exhaustion (memory, file handles, connections) not mitigated
  • Inconsistent error responses revealing application internals

Attack Examples

POST /api/upload â€” Content-Length: 999999999999 Server: java.lang.OutOfMemoryError â†’ full stack trace + internal paths // Application crashes, denying service to all users

Oversized request causes unhandled memory exhaustion, crashing the server and leaking internal diagnostic information.

Remediation

  • Implement global error handlers with generic user-facing messages
  • Set resource limits (max upload size, connection pools, timeouts)
  • Use error boundaries and circuit breakers for graceful degradation
  • Test with fuzzing, edge cases, and chaos engineering
